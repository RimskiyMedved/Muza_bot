# Тест-кейсы для Telegram-бота бронирований «Муза»

Документ — чек-лист для ручного тестирования и основа для автотестов.

## 0) Подготовка тестовой среды

### 0.1 Конфигурация
- **Предусловия**
  - В `.env` заданы: `TELEGRAM_BOT_TOKEN`, `ADMIN_CHAT_ID`, `SPREADSHEET_ID`, `GOOGLE_CREDENTIALS_PATH`, `SHEET_NAME`, `FREE_SHEET_NAME`, `LEADS_SHEET_NAME`.
  - Сервисному аккаунту выдан доступ «Редактор» к таблице.
- **Проверка**
  - Запустить `python bot.py`.
  - В логах видно: токен задан, `SPREADSHEET_ID` задан, `credentials` найден.
- **Ожидаемо**
  - Бот стартует без исключений.

### 0.2 Подготовка данных Google Sheets
Создайте/подготовьте листы и данные:
- **Лист `Бронирования`**: заголовки как в `sheets.py` `HEADERS_BOOKINGS` и 2–3 записи:
  - одна на будущую дату (занято),
  - одна на сегодня,
  - одна в прошлом.
- **Лист `Свободные`**: заголовки `HEADERS_FREE` и 10–20 будущих дат (включая те, которых нет в «Бронирования»).
- **Лист `Авито`**: можно пустой (бот сам проставит заголовки при первом лиде).

## 1) Авто-приветствие и авто-режим (сообщения в чате)

### 1.1 Приветствие нового пользователя (не админ)
- **Шаги**
  - Новый пользователь (не из `ADMIN_CHAT_ID`) пишет любое сообщение без даты (например: «привет»).
- **Ожидаемо**
  - Бот отправляет `GREETING_TEXT`.
  - Дальше бот не отвечает (т.к. дата не распознана).

### 1.2 Приветствие не повторяется
- **Шаги**
  - Тот же пользователь пишет снова «привет».
- **Ожидаемо**
  - Бот НЕ отправляет `GREETING_TEXT` повторно.

### 1.3 Админ не получает авто-приветствие
- **Шаги**
  - Пользователь-админ пишет любое сообщение.
- **Ожидаемо**
  - Бот не отправляет `GREETING_TEXT`.

### 1.4 Сообщение с датой в первом сообщении + приветствие
- **Шаги**
  - Новый не-админ пишет: «15 мая хочу зал».
- **Ожидаемо**
  - Бот отправляет приветствие.
  - Затем (в этой же обработке) проверяет дату и отвечает занято/свободно.

## 2) FAQ (цена / вместимость)

### 2.1 Вопрос про цену (ключевые слова)
- **Шаги**
  - Пользователь пишет: «Сколько стоит аренда?» / «цена» / «прайс».
- **Ожидаемо**
  - Ответ `FAQ_PRICE` (Markdown).
  - Никаких проверок даты (если даты в тексте нет).

### 2.2 Вопрос про вместимость
- **Шаги**
  - Пользователь пишет: «Сколько человек вмещает?» / «вместимость».
- **Ожидаемо**
  - Ответ `FAQ_CAPACITY` (Markdown).

### 2.3 FAQ имеет приоритет над датой
- **Шаги**
  - Пользователь пишет: «цена на 20.05?».
- **Ожидаемо**
  - Отвечает `FAQ_PRICE` и НЕ проверяет дату (текущая логика — FAQ раньше даты).

## 3) Парсинг даты (`parse_date`)

### 3.1 Формат `dd.mm.yyyy`
- **Данные**: «20.05.2026»
- **Ожидаемо**
  - Распознана дата 20.05.2026.

### 3.2 Формат `dd.mm` (без года) с будущим переносом
- **Предусловия**
  - Сегодня позже 20.05 текущего года.
- **Шаги**
  - Написать «20.05».
- **Ожидаемо**
  - Дата распознана как 20.05 следующего года.

### 3.3 Формат `yyyy-mm-dd`
- **Шаги**
  - «2026-05-20».
- **Ожидаемо**
  - Дата распознана.

### 3.4 Формат `dd <месяц_словом>` (русский)
- **Шаги**
  - «20 мая», «20 май», «20 мая 2026».
- **Ожидаемо**
  - Распознана корректная дата.

### 3.5 Невалидная дата
- **Шаги**
  - «31.02.2026», «99.99».
- **Ожидаемо**
  - Дата не распознана (`None`), бот не отвечает в авто-режиме.

### 3.6 Шум вокруг даты
- **Шаги**
  - «Хочу 20.05.2026, гостей 50».
- **Ожидаемо**
  - Дата распознана.

## 4) Авто-проверка даты (занято/свободно)

### 4.1 Дата занята
- **Предусловия**
  - В «Бронирования» есть запись на тестовую дату.
- **Шаги**
  - Пользователь пишет дату.
- **Ожидаемо**
  - Ответ `BUSY_TEXT`.

### 4.2 Дата свободна
- **Предусловия**
  - В «Бронирования» НЕТ записи на эту дату.
- **Шаги**
  - Пользователь пишет дату.
- **Ожидаемо**
  - Ответ `FREE_TEXT`.
  - Пользователь попадает в `_waiting_contact`.

### 4.3 Повторное сообщение с датой от пользователя в ожидании контакта
- **Предусловия**
  - Пользователь уже получил `FREE_TEXT`, и бот ждёт контакт.
- **Шаги**
  - Пользователь пишет снова дату (без телефона/ника).
- **Ожидаемо**
  - Бот снова ответит `FREE_TEXT` (и останется ждать контакт).

### 4.4 Получение контакта после свободной даты (телефон)
- **Предусловия**
  - Пользователь в `_waiting_contact`.
- **Шаги**
  - Пользователь отправляет: `+7 999 123-45-67` или `89991234567`.
- **Ожидаемо**
  - Бот отправляет сообщение `@rimskiymedved` в тот же чат.
  - Пользователь удаляется из `_waiting_contact`.

### 4.5 Получение контакта после свободной даты (ник)
- **Шаги**
  - Пользователь отправляет: `@someNick`.
- **Ожидаемо**
  - Аналогично: тег `@rimskiymedved`, ожидание снимается.

### 4.6 Контакт не распознан
- **Шаги**
  - Пользователь отправляет: «мой номер 12345» или «напишите мне тут».
- **Ожидаемо**
  - Тег не отправляется, ожидание остаётся.

### 4.7 Ошибка доступа к таблице при `check_date`
- **Предусловия**
  - Неверный `SPREADSHEET_ID` или нет доступа сервисного аккаунта.
- **Шаги**
  - Пользователь пишет дату.
- **Ожидаемо**
  - Бот не падает; в логах ошибка.
  - (Текущее поведение: пользователю ничего не отвечает.)

## 5) Авито-парсер и лиды

### 5.1 Распознавание заголовка Авито
- **Шаги**
  - Отправить сообщение, содержащее строку вида:
    - `Собеседник: Иван [#user123] ... [#adv456]`
    - и далее блок с `Локация:` и текстом после него.
- **Ожидаемо**
  - Бот не отвечает в чат.
  - В логах фиксируется распознавание.
  - `_avito_users` запоминает `#user123 → Иван`.

### 5.2 Извлечение контента после `Локация:`
- **Шаги**
  - В авито-сообщении после `Локация:` указать несколько строк текста.
- **Ожидаемо**
  - В обработку лида уходит только текст после `Локация:`.

### 5.3 Лид сохраняется при наличии телефона
- **Шаги**
  - В контенте указать телефон в формате `+7...` или `8...` или `11 цифр`.
- **Ожидаемо**
  - В лист `Авито` добавляется строка: дата/время, имя, телефон, пустой ник, объявление (`Авито #<id>` если был `adv`).

### 5.4 Лид сохраняется при наличии ника
- **Шаги**
  - В контенте указать `@nickname` (>= 4 символов после `@`).
- **Ожидаемо**
  - В лист `Авито` добавляется строка: ник заполнен.

### 5.5 Лид НЕ сохраняется без телефона и ника
- **Шаги**
  - В контенте нет телефона и `@...`.
- **Ожидаемо**
  - Запись в `Авито` не добавляется.

### 5.6 Ошибка записи лида (нет доступа к листу)
- **Предусловия**
  - У сервисного аккаунта нет доступа или неверное имя листа.
- **Шаги**
  - Отправить валидное авито-сообщение с телефоном.
- **Ожидаемо**
  - Бот не падает; ошибка логируется.

## 11) Диалог `/add` (только админ)

### 11.1 Не-админ не может `/add`
- **Ожидаемо**
  - «Команда только для администратора.», диалог завершается.

### 11.2 Нераспознанная дата
- **Шаги**
  - Админ: `/add`
  - Ответить: «завтра» / «когда-нибудь».
- **Ожидаемо**
  - Просит дату снова, остаётся в состоянии ввода даты.

### 11.3 Дата уже занята (предупреждение)
- **Предусловия**
  - Дата есть в «Бронирования».
- **Шаги**
  - Ввести эту дату.
- **Ожидаемо**
  - Предупреждение, что дата занята, но можно продолжить.

### 11.4 Успешное добавление новой брони
- **Шаги**
  - Ввести дату (свободную), гостей, имя, телефон.
  - На шаге «Источник» выбрать кнопку из `KB_SOURCE`.
  - На шаге типа клиента выбрать кнопку из `KB_TYPE`.
  - Комментарий: «—».
- **Ожидаемо**
  - В «Бронирования» появляется/обновляется строка с weekday.
  - Из «Свободные» дата удаляется (если была).
  - Бот пишет карточку с данными, комментарий отображается «—».

### 11.5 Комментарий пустой/нет
- **Шаги**
  - На комментарии ввести: `нет` / `-` / `н/а` / `.`.
- **Ожидаемо**
  - В таблицу пишется пустой комментарий.

### 11.6 `guests` как текст
- **Шаги**
  - Ввести «до 80».
- **Ожидаемо**
  - В таблицу попадёт «80» (текущая логика берёт первую цифру).

### 11.7 Отмена диалога `/stop`
- **Шаги**
  - Запустить `/add`, дойти до любого шага, отправить `/stop`.
- **Ожидаемо**
  - «Отменено.», клавиатура убрана, `user_data` очищен.

### 11.8 Ошибка записи в таблицу
- **Предусловия**
  - Сломать доступ к таблице.
- **Ожидаемо**
  - Сообщение «Ошибка при записи: …», диалог завершается.
