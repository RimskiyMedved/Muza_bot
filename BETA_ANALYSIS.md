## 🔴 КРИТИЧЕСКИЕ БАГИ (ломают функциональность)

### 1. Нельзя удалить пользователя-администратора до первого входа

**Файл:** `database.py:392`, `api.py` `/api/admin/users DELETE`

Когда суперадмин добавляет пользователя через Панель (`POST /api/admin/users`), в базу записывается `telegram_id = NULL` — потому что пользователь ещё не написал боту.

Функция удаления:
```python
def remove_allowed_user(telegram_id: int) -> bool:
    con.execute("DELETE FROM allowed_users WHERE telegram_id = ?", (telegram_id,))
```

В SQL `NULL != 0`, `NULL != anything` — запрос всегда возвращает 0 строк. Пользователя невозможно удалить, пока он не напишет боту хотя бы раз.

**Эффект:** Если добавить пользователя по ошибке — его нельзя убрать через UI. Только руками через sqlite3 на сервере.

**Фикс:** В `remove_allowed_user` добавить удаление по `username` как fallback:
```python
con.execute(
    "DELETE FROM allowed_users WHERE telegram_id = ? OR (telegram_id IS NULL AND id = ?)",
    (telegram_id, telegram_id)
)
```
И в API передавать `id` записи, а не `telegram_id`.

### 2. `sync_from_sheets` может полностью стереть базу

**Файл:** `database.py:108`

```python
con.execute("DELETE FROM bookings")
for b in bookings:
    con.execute("INSERT INTO bookings ...")
```

Сначала удаляет ВСЕ брони, потом вставляет новые. Если Google Sheets API вернул пустой список (сетевой сбой, quota exceeded, временная ошибка) — вся база бронирований уничтожена. Нет никакой проверки `if len(bookings) > 0`.

**Эффект:** После рестарта бота при нестабильном интернете — все брони исчезают из SQLite. Бот начнёт говорить что все даты свободны.

**Фикс:**
```python
if not bookings:
    log.warning("⚠️ Sheets вернул 0 бронирований — пропускаем sync во избежание потери данных")
    return
```

### 3. Двойная запись в SQLite при каждом действии через Mini App

**Файлы:** `api.py:287-292`, `sheets.py:277-283`

При создании брони через API:
1. `api.py` вызывает `_sheets_write("add", ...)` → внутри это `sheets.add_booking()`
2. `sheets.add_booking()` **сама** пишет в SQLite: `_db.upsert_booking(...)`
3. Потом `api.py` **ещё раз** вызывает `database.upsert_booking(...)`

То же самое при удалении: `sheets.remove_booking()` → `_db.delete_booking()`, и потом `api.py` снова `database.delete_booking()`.

**Эффект:** Пока не ломает ничего (upsert идемпотентен), но это замаскированная бомба — если логика разойдётся, будет непредсказуемое поведение. Уже сейчас `changed_at` и `synced_at` в SQLite перезаписываются с разницей в миллисекунды.

### 5. Приветственное сообщение отправляется администраторам

**Файл:** `bot.py:444`

```python
if user.id not in _seen_users and user.id not in ADMIN_IDS:
```

`ADMIN_IDS` — это `{SUPERADMIN_ID}` (только суперадмин). Пользователи из `allowed_users` (остальные три женщины) при первом сообщении получат приветствие «Здравствуйте! Я помогу проверить свободные даты» — как будто они клиент, а не администратор.

**Фикс:**
```python
if user.id not in _seen_users and not is_admin(update):
```

### 6. FAQ по цене блокирует проверку даты в том же сообщении

**Файл:** `bot.py:437-443`

Порядок проверок в `auto_check`:
1. Цена/стоимость → FAQ
2. Вместимость → FAQ
3. ...дата

Если клиент пишет **«сколько стоит 15 июня?»** — бот ответит прайсом и НЕ проверит дату. Клиент не узнает, свободна ли 15 июня.

**Фикс:** Сначала парсить дату, потом проверять FAQ. Или делать оба ответа если в сообщении и дата, и вопрос о цене.

### 7. Двойное бронирование: предупреждение без блокировки

**Файл:** `bot.py:1110-1122`

При `/add`, если дата уже занята:
```python
await update.message.reply_text(
    f"⚠️ {fmt_date(d)} уже занята.\n"
    "Продолжайте чтобы перезаписать — или нажмите ❌ Отмена.",
)
```
Состояние продолжается в `Add.GUESTS` — никакого подтверждения не требуется. Достаточно проигнорировать предупреждение и продолжить заполнять — бронь перезапишется без явного «Да, перезаписать».

### 8. `_seen_users` и `_waiting_contact` — in-memory, сбрасываются при рестарте

**Файл:** `bot.py:378-379`

```python
_seen_users:      set[int] = set()
_waiting_contact: set[int] = set()
```

При каждом `docker compose restart muza_bot` все уже виденные пользователи снова получат приветствие. Для активных клиентов это выглядит странно.

Критичнее: `_waiting_contact` — **мёртвый код**. Он объявлен, но нигде не используется в логике. Изначально, видимо, задумывался для отслеживания клиентов после ответа «дата свободна» — но логика не реализована.

## 🟡 СРЕДНИЕ ПРОБЛЕМЫ

### 9. Docstring в `get_calendar` стоит ПОСЛЕ кода — не работает как docstring

**Файл:** `api.py:177-183`

```python
async def get_calendar(...):
    # Логируем открытие
    try:
        database.log_access(...)
    except Exception:
        pass
    """
    Возвращает статус каждого дня месяца.  ← это строковый литерал, не docstring
    """
```

Python распознаёт docstring только если строка — первое выражение функции. Здесь это просто выражение без эффекта. Не ломает ничего, но вводит в заблуждение.

### 10. `log_access` пишет запись при КАЖДОМ запросе календаря

**Файл:** `api.py:177`

Каждый раз когда пользователь листает месяц в Mini App — пишется строка в `access_log`. При переключении 10 месяцев = 10 записей. Таблица растёт быстро. Лимит в `get_access_log` = 200 строк, но сама таблица не очищается.

---

### 11. WAL-режим SQLite включается только в API, не в боте

**Файл:** `api.py:90-96`, `database.py:init_db()`

```python
# api.py — включает WAL
con.execute("PRAGMA journal_mode=WAL")
```

В `bot.py` при `init_db()` WAL не устанавливается. WAL — это настройка файла базы, она сохраняется между запусками, но только если API стартовал первым. Если запустить только бота — WAL не включится, и конкурентные записи (бот + апи одновременно) могут конфликтовать.

### 12. `_normalize_phone` продублирована в двух файлах

**Файлы:** `database.py:185`, `sheets.py:327`

Идентичный код в двух местах. Если логика нормализации изменится — нужно менять в двух местах.

### 13. CORS открыт полностью

**Файл:** `api.py:52-56`

```python
app.add_middleware(CORSMiddleware, allow_origins=["*"], ...)
```

Любой сайт в интернете может делать запросы к API. Защита только через `X-Init-Data`, но это означает что при утечке токена/initData — атаку можно провести с любого сайта. Лучше ограничить до доменов Telegram: `https://*.telegram.org` и ngrok-домена.

---

### 14. initData в Mini App истекает через 24 часа — молча

**Файл:** `api.py:121`

```python
if time.time() - auth_date > 86400:
    raise HTTPException(401, "initData expired")
```

Если кто-то откроет Mini App, уйдёт, и вернётся через сутки — все запросы будут падать с 401. В `webapp/index.html` нет обработки этого кейса с понятным сообщением пользователю.

### 15. Статистика: `by_month` всегда показывает будущие месяцы даже без броней

**Файл:** `api.py:460-471`

```python
if by_month.get(key, 0) > 0 or delta >= 0:
```

Условие `delta >= 0` означает что текущий месяц и все следующие 11 месяцев всегда попадут в график — даже если там 0 броней. График растянут на год вперёд независимо от данных.

## 🟢 МЕЛКИЕ ЗАМЕЧАНИЯ

### 16. Источники рекламы жёстко зашиты в код

**Файл:** `bot.py:121-133`

```python
KB_SOURCE = ReplyKeyboardMarkup([
    ["Ангелина", "Ведвед"],
    ["Горько", "Даша рассылка"],
    ...
])
```

Если состав команды изменится — нужен деплой. Лучше хранить в `.env` или SQLite, чтобы суперадмин мог редактировать через Панель.

### 17. `add_phone` — нет никакой валидации

**Файл:** `bot.py:1140-1143`

Телефон сохраняется как есть. Можно ввести «нет», «—», или вообще случайный текст. Это попадёт в Sheets и в статистику.

### 18. `cmd_stats` в боте и `/api/stats` считают по-разному

В боте `/stats` читает из Google Sheets напрямую (`get_all_bookings` из `sheets.py`). В Mini App — из SQLite. Если синхронизация не прошла — данные расходятся. Нет индикации «последняя синхронизация была N минут назад».

---

### 19. В логах старт бота показывает `ADMIN_IDS: {45028744}` — вводит в заблуждение

**Файл:** `bot.py:1477`

```python
log.info(f"  ADMIN_IDS      : {ADMIN_IDS or '⚠️  НЕ ЗАДАНЫ'}")
```

`ADMIN_IDS` — это только суперадмин. Реальные администраторы из SQLite не показываются при старте. Кто смотрит логи думает что только один человек имеет доступ.
